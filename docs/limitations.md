# Limitations

DLP Inspector 的定位是轻量级保密审计自查与报告归档原型。它适合用于授权范围内的预检查、教学演示和内部审计线索发现，不应被描述为企业级实时 DLP 平台。

## 规则匹配边界

- 当前核心识别方式以关键词和正则匹配为主。
- 规则匹配可能产生误报，例如业务文本中出现“保密”等普通描述。
- 规则匹配也可能漏报，例如敏感信息被改写、截图化、压缩、加密、拆分或使用未配置的表达方式。
- 扫描结果应作为人工复核线索，不应直接等同于最终违规结论。

## Web 扫描边界

- Web 扫描基于 `requests` 获取页面内容，并使用 `BeautifulSoup` 解析静态 HTML。
- 爬取范围限制在入口 URL 的同域名下，并通过最大深度控制扫描范围。
- 当前实现不渲染 JavaScript，不执行浏览器自动化，不处理登录态复杂交互，也不保证覆盖前端动态加载的数据。
- 网络错误、重定向、反爬策略、内容类型限制和编码识别都可能影响扫描结果。

## 数据库审计边界

- 数据库模块当前面向 MySQL，使用用户提供的标准连接凭证访问目标库。
- 扫描逻辑只枚举数据表并提取文本类型字段，包括 `varchar`、`text`、`char`、`longtext`、`mediumtext`、`tinytext`。
- 本项目不做数据库漏洞扫描，不检测弱口令、SQL 注入、越权访问、配置风险、补丁状态或权限模型缺陷。
- 对大规模数据库，当前实现可能需要分页、限速或增量扫描优化。

## OCR 边界

- 图片模块依赖 `PaddleOCR`，当前实现默认 `use_gpu=False`，即 CPU 推理。
- OCR 命中效果受图片分辨率、清晰度、压缩质量、字体、方向、遮挡、版式和模型环境影响。
- 项目不包含自研 OCR 模型；模型文件和缓存应按实际安装或打包方式处理。
- OCR 依赖较重，部署和打包时需要额外验证 PaddlePaddle、OpenCV、模型缓存与运行平台兼容性。

## Office 解析边界

- 新版 Office 文档主要依赖 `python-docx`、`python-pptx`、`openpyxl`、`PyMuPDF` 等库解析。
- 旧版 `.doc` / `.ppt` 文件解析依赖 Windows COM 和本机 Office 组件，跨平台能力有限。
- 加密、损坏、受权限保护或格式伪装的文档可能无法读取，只能记录为异常或待人工复核对象。

## 产品边界

- 本项目不提供实时外发阻断、内核级文件拦截、网络流量检测、终端策略下发、多用户权限中心或集中管控平台。
- 它解决的是敏感内容发现、证据留存和 Excel 报告归档问题，不解决企业级 DLP 的实时控制问题。
